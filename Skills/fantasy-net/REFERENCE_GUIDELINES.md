# fantasy-net Reference 编写规范

这份规范用于后续继续给 `fantasy-net` skill 增加或重构 reference 文档，目标是：

- 不丢功能覆盖
- 尽量节省 token
- 让模型优先读取最小必要上下文
- 让文档结构长期保持一致

## 总原则

### 1. 先做入口页，再做细节页

大主题不要一上来就把所有内容堆进一个文档。

优先结构：

```text
大主题入口
  -> 实现
  -> 建模 / 选择
  -> 排错
  -> 模板 / 场景
```

入口页负责：

- 判断用户当前属于哪类问题
- 给出最短 Workflow
- 列 3-5 条必须记住的规则
- 指向更具体的子文档

入口页不要承担：

- 全量 API 手册
- 大量示例堆叠
- 所有边界条件的展开解释

### 2. 让模型按问题类型读取，而不是按主题整包读取

用户问“怎么实现”时，不应该被迫读完整的“实现 + 建模 + 排错 + 原理”。

优先拆法：

- `index.md` / 入口页
- `*-implement.md`
- `*-design.md`
- `*-troubleshoot.md`
- `*-scenarios.md`

不是每个主题都必须有这几类，但大主题优先按这个思路设计。

### 3. 不为了拆而拆

如果一个文档同时满足以下条件，就不要强拆：

- 单主题
- 结构清晰
- 读它时通常整篇都需要
- 长度还在合理范围

经验判断：

- 50-120 行：通常已经很好
- 120-180 行：看是否混了多个任务类型
- 180+ 行：优先检查是否值得入口化或拆分

### 4. 模板和参考手册不要放进入口页

像 `templates/Fantasy.config` 这类完整模板，应该保留为独立文件。

入口页只说：

- 什么时候需要去读模板
- 模板解决什么问题

不要把模板内容再复制一遍到入口页里。

### 5. SKILL.md 的description 一定要补全上关键词用来提高命中

## 推荐文档类型

### 入口页

适用：

- 一个主题下面已经有 2 个以上子文档
- 或者一个主题天然有多种问题路径

推荐结构：

```markdown
# Xxx 入口

一句话说明这个主题处理什么问题。

## Workflow

```text
问题 A -> a.md
问题 B -> b.md
问题 C -> c.md
```

## 必记规则

1. ...
2. ...
3. ...

## 子文档

- `a.md` - ...
- `b.md` - ...
```

### 实现页

适用：

- 用户已经知道要用什么机制
- 需要直接落代码或配置

应该包含：

- 最短步骤
- 最小示例
- 关键 API
- 最少但足够的注意事项

不要包含：

- 大段背景解释
- 跟实现无关的排错内容

### 建模页

适用：

- 用户问题不是“怎么写”，而是“挂在哪”“选哪种方式”“什么时候建会话实体”

应该包含：

- 选择规则
- 推荐结构
- 反例
- 设计边界

### 排错页

适用：

- 用户给现有代码
- 用户问“为什么不工作”

应该包含：

- 常见错误
- 排查顺序
- 最短对照示例

排错页尽量写成 checklist，而不是长篇教程。

### 场景页

适用：

- 主题常被用户按“任务场景”提问，而不是按 API 提问

例如：

- 新建项目
- 新增 Scene
- 改数据库
- 多区服

## 命名规范

优先使用可预测命名：

- `index.md`：总入口
- `xxx-implement.md`：实现
- `xxx-design.md`：建模 / 选择
- `xxx-troubleshoot.md`：排错
- `xxx-scenarios.md`：按场景操作

避免使用含糊命名：

- `advanced.md`
- `misc.md`
- `extra.md`
- `more.md`

## 内容编写规则

### 1. 先告诉模型“什么时候读这个文件”

每个文档开头优先写：

- 这个文件适合什么问题
- 什么情况下不要读它

### 2. 优先写决策信息，不优先写百科信息

对 skill 来说，最重要的是让模型快速做出正确选择。

优先写：

- 什么时候用
- 什么时候不用
- 先做哪一步
- 哪些规则不能错

其次才是：

- 详细背景
- 完整原理
- 扩展说明

### 3. 示例要短、能直接迁移

示例尽量：

- 20-40 行内
- 展示一个完整动作
- 尽量贴近 Fantasy 实际命名风格

避免：

- 超长示例把文档主干冲掉
- 同一概念给 5 个几乎重复的例子

### 4. 尽量写“规则”，少写“散文式说明”

更推荐：

- Workflow
- 必记规则
- 场景分流
- 检查清单

少用长段说明去覆盖多个问题类型。

### 5. 解释“为什么”，但要短

不是完全只写结论。

应该短解释：

- 为什么推荐这样做
- 为什么不要那样做

但不要写成长篇设计文。

## 更新现有文档时的检查清单

每次新增或重构 reference 后，至少检查下面这些点：

1. `SKILL.md` 的导航描述是否同步
2. 入口页描述是否还是“入口”，没有写成“完整手册”
3. 子文档之间的相互引用是否存在
4. 是否出现已经删除的旧文件名
5. 是否把模板、实现、排错重新混回一个文件
6. 文档是否真的减少了高频问题的读取范围

## 什么时候值得继续拆分

满足任一情况时，优先考虑继续拆：

- 一个文档同时覆盖“实现 + 建模 + 排错”
- 模型为了解决一个简单问题必须读 150+ 行
- 文档里的不同部分明显只服务于不同类型的问题
- 同类问题已经在多个地方重复出现

## 什么时候不要继续拆分

满足大部分情况时，不要再拆：

- 文档长度已经不大
- 单主题且使用时通常整篇都需要
- 继续拆只会增加跳转次数
- 拆分后文件名会变得难记

## 适合 fantasy-net 的长期结构

后续新增大主题时，优先复用现在已经验证有效的模式：

- ECS：入口 + 定义 + 操作 + 对象池 + 生命周期
- Event：入口 + EventAwaiter 入口 + 实现 / 建模 / 排错
- Protocol：入口 + Outer / Inner / Common / Export
- Roaming：入口 + Handler 入口 + Messaging / Transfer
- Config：入口 + 场景 + 模板
- Unity：入口 + Setup / Connection / Session / Handler

新主题尽量也长成这个样子，而不是回到“一篇大杂烩”。

## 一句话标准

如果一个用户问题只需要这套主题里的 20% 信息，那模型就应该只读到那 20%，而不是整包全读。
